## Design Context

### Users
Small dev teams (2–10 developers) who need to share environment variables. They're already comfortable with .env files and terminal workflows. They use this during active development, in their IDE, alongside their terminal.

### Brand Personality
Direct, collaborative, effortless. Handoff feels like a tool a thoughtful coworker built, not enterprise software. It's the pocket knife, not the Swiss Army toolbox.

### Aesthetic Direction
- **Tone**: Warm utilitarian. Clean like good documentation, approachable like a well-designed indie tool.
- **Theme**: Light (primary), dark mode supported. Developers browse landing pages during work hours.
- **Palette**: Warm amber direction (oklch hue ~70). Paper-white backgrounds, warm darks, amber accents used sparingly. All neutrals tinted toward the brand hue.
- **Typography**: Schibsted Grotesk (display, 700-800) + Figtree (body, 300-600). Direct Scandinavian character paired with warm geometric readability.
- **Anti-references**: Enterprise secrets platforms (Vault, Doppler), AI-default dark+neon aesthetic, overly corporate blues/purples.
- **References**: Linear's clarity, Vercel's confidence, but warmer and more approachable than either.

### Design Principles
1. **Simplicity is the product**: if it feels complex, we've failed. Every design choice should reinforce "this is easy."
2. **Developer-native**: use code/terminal idioms naturally (monospace for actual code, dark panels for env displays), but don't costume the whole UI as "techy."
3. **Warmth over cold precision**: tinted neutrals, paper-like backgrounds, amber accents. Inviting, not clinical.
4. **Type-driven hierarchy**: use size, weight, and color to create hierarchy. No decorative flourishes needed.
5. **Show, don't tell**: the env file display and terminal mocks communicate the product better than paragraphs of copy.
